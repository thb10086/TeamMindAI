# AI 企业大脑平台：产品原型文档 v1.0

> 本文档基于《AI 企业大脑平台：系统 SPEC 规划文档 v1.0》和《项目架构设计文档 v1.0》编写，用于指导 UI、前端、后端和产品落地。本文档以页面结构、功能布局、核心交互、字段内容、状态流转和 MVP 范围为重点，目标是让团队可以基于本文档直接开始设计原型和开发页面。

> **【实现说明】** 页面与交互设计与具体技术无关，依然有效。实际实现采用 **统一 TypeScript 全栈**：Next.js 15 (App Router) + React 19 + Tailwind CSS v4 + shadcn/ui（new-york, neutral）+ lucide-react。导航/布局/组件已在代码中按本文 §3 信息架构落地（侧边栏一级导航、AI 对齐室、工作台等）。详见《AI企业大脑平台_04_技术栈落地说明.md》与仓库 `README.md`。

---

## 1. 产品原型目标

AI 企业大脑平台的产品原型必须围绕一个核心闭环设计：

```text
想法输入 → AI 需求澄清 → 标准需求卡片 → 需求评审 → AI 任务拆解 → 看板推进 → 通知提醒 → 项目记忆沉淀
```

原型设计不能只做成普通项目管理系统，而要突出 AI 员工的参与感、项目上下文的连续性、需求对齐的流程感和任务推进的可追踪性。

---

## 2. 设计原则

### 2.1 先对齐，再执行

任何需求进入开发前，页面都要引导用户补全背景、目标、范围、验收标准。

### 2.2 AI 是员工，不是聊天框

AI 不应该只以聊天框出现，而应该在需求、任务、会议、报告、通知等场景中主动承担职责。

### 2.3 项目空间是核心容器

所有需求、任务、会议、决策、知识、AI 对话都必须归属到某个项目。

### 2.4 信息要结构化

AI 输出结果要能够落到需求卡片、任务卡片、决策记录、会议纪要、项目记忆中。

### 2.5 关键节点需要人工确认

AI 可以生成草案，但需求确认、任务分配、邮件群发、上线验收等关键动作必须支持人工确认。

## 4. 用户角色与界面关注点

### 4.1 老板 / 负责人

首页优先展示：项目数量、延期任务、阻塞风险、本周进展、AI 管理建议、项目健康评分。

### 4.2 项目负责人

优先关注：需求状态、任务进度、人员负载、待确认事项、风险预警、周报。

### 4.3 技术负责人

优先关注：需求技术影响、任务拆解、接口设计、阻塞问题、技术风险。

### 4.4 开发人员

优先关注：我的任务、任务背景、验收标准、关联需求、截止时间、阻塞反馈。

### 4.5 运营 / 销售 / 客服

优先关注：需求提交入口、客户反馈、需求状态、是否被采纳、预计排期。

# 5.6 AI 对齐室

## 页面目标

让用户把模糊想法提交给 AI 产品经理，由 AI 基于项目上下文进行澄清和标准化。

## 页面结构

```text
┌─────────────────────────────────────────────────────────────┐
│ 左侧：项目上下文                                             │
│ - 项目目标                                                   │
│ - 历史需求                                                   │
│ - 相关决策                                                   │
│ - 相关知识                                                   │
├───────────────────────┬─────────────────────────────────────┤
│ AI 员工列表             │ 对话区                               │
│ ○ AI 产品经理           │ 用户：我想做一个客户工单智能分析       │
│ ○ AI 项目经理           │ AI：我需要先确认几个问题……             │
│ ○ AI 架构师             │                                     │
│ ○ AI 测试               │ [输入框] [发送] [生成需求卡片]          │
├───────────────────────┴─────────────────────────────────────┤
│ 右侧/下方：AI 生成的需求草案                                  │
│ [保存为需求卡片] [继续澄清] [发起评审]                         │
└─────────────────────────────────────────────────────────────┘
```

## 核心能力

```text
选择 AI 员工
查看 AI 引用的项目上下文
AI 自动追问
生成需求草案
保存为需求卡片
将对话沉淀为项目记忆
```

## 关键交互

1. 用户输入一句话想法。
2. AI 产品经理判断信息是否足够。
3. 如果不足，AI 生成澄清问题。
4. 用户补充回答。
5. AI 生成标准需求卡片。
6. 用户确认后保存。

## 需求草案结构

```text
需求名称
业务背景
当前问题
目标用户
业务目标
功能范围
不做范围
用户故事
验收标准
优先级建议
待确认问题
```


# 5.23 界面设计（可交互低保真原型）

> 该页属于「项目空间」二级导航，**不在一级导航**。入口：项目空间 → 界面设计 tab。

## 页面目标

把「已确认需求」转成**可点击的低保真界面原型**（真实可交互页面，而非文字描述），供团队走查、对齐与设计师细化。

## 信息架构

```text
列表视图（项目空间 → 界面设计 tab）：
- 本项目全部设计稿卡片（状态 / 界面数 / 版本 / 关联需求）
- 「从需求发起界面设计」选择器：选已确认需求 → 创建设计稿

画布视图（/design/[id]）：
- 左侧：界面（屏）列表，含每屏生成状态（待生成 / 生成中 / 已完成）
- 中间：沙箱预览（桌面 / 移动视口切换）
- 顶部：AI 生成 / 继续生成 / 重新生成、进度
```

## 生成流程（人在回路 + 进度可见）

```text
选择已确认需求 → 发起界面设计（创建设计稿）
  ↓
AI 规划信息架构与界面清单（屏）
  ↓
逐屏生成可交互 HTML（实时进度：已生成 N / 总 M，每屏完成即可查看）
  ↓
每屏即时入库：切走 / 刷新后已生成的保留，未完成可「继续生成」续跑
  ↓
全部完成后沉淀为项目记忆
```

## 关键交互

```text
- 界面用语义化 HTML + Tailwind 原子类渲染，含中文示例数据与关键状态（空态 / 加载 / 错误以文案呈现）。
- 界面之间用 data-goto 衔接：点击界面内按钮 / 卡片可在画布内跳转到目标界面，形成可走查的用户流程。
- 预览运行在沙箱 iframe（allow-scripts，无 allow-same-origin），与主应用隔离。
- 这是低保真结构与交互原型：克制配色（中性灰白），不追求最终视觉。
```

## 状态流转

```text
DRAFT(AI 草案) → DESIGNING(设计中) → IN_REVIEW(评审中) → APPROVED(已定稿) → ARCHIVED(已归档)
```

## 12. 产品原型总结

AI 企业大脑平台的产品原型不是围绕“录入任务”设计，而是围绕“对齐信息”设计。

页面设计的主线必须始终保持：

```text
把一句话想法变成清晰需求；
把清晰需求变成可执行任务；
把任务执行变成可追踪周期；
把执行过程变成企业长期记忆。
```

第一版只要把“项目空间 + AI 需求澄清 + 需求卡片 + 任务看板 + 邮件通知 + 项目记忆”跑通，就已经能解决初创公司内部最核心的信息差和需求不清问题。
